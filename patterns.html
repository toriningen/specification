5. Patterns¶

5.1. Patterns¶

Syntax

Pattern ::=
    |? PatternWithoutAlternation (| PatternWithoutAlternation)*

PatternList ::=
    Pattern (, Pattern)+ ,?

PatternWithoutAlternation ::=
    PatternWithoutRange
  | RangePattern

PatternWithoutRange ::=
    IdentifierPattern
  | LiteralPattern
  | MacroInvocation
  | ParenthesizedPattern
  | PathPattern
  | ReferencePattern
  | RestPattern
  | SlicePattern
  | StructPattern
  | TuplePattern
  | TupleStructPattern
  | WildcardPattern

Legality Rules

5.1:1 A pattern is a construct that matches a value which satisfies all the criteria of the pattern.

5.1:2 A pattern-without-alternation is a pattern that cannot be alternated.

5.1:3 A subpattern is a pattern nested within another pattern.

5.1:4 A pattern has a type, with the exception of the rest pattern if it is not the inner pattern of a slice pattern or the pattern of a possibly nested identifier pattern of a slice pattern.

5.1:5 The expected type of a pattern is the type of the value the pattern is being matched against.

5.1:6 Any two pattern-without-alternations that are or-ed using character 0x7C (vertical line) are subject to the following restrictions:

  • 5.1:7 The types of the two pattern-without-alternations shall be unifiable.

  • 5.1:8 The bindings of the two pattern-without-alternations shall be the same, shall have unifiable types, and shall have the same binding modes.

5.1.1. Identifier Patterns¶

Syntax

IdentifierPattern ::=
    ref? mut? Binding BoundPattern?

BoundPattern ::=
    @ Pattern

Legality Rules

5.1.1:1 An identifier pattern is a pattern that binds the value it matches to a binding.

5.1.1:2 A bound pattern is a pattern that imposes a constraint on a related identifier pattern.

5.1.1:3 An identifier pattern yields a binding. An identifier pattern with keyword mut yields a mutable binding.

5.1.1:4 The identifier pattern enters its binding into pattern scope in the value namespace if it does not resolve to a constant.

5.1.1:5 It is a static error if the identifier pattern consists of anything other than a binding when the binding resolves to a constant.

5.1.1:6 An identifier pattern is irrefutable when:

  • 5.1.1:7 It has a bound pattern and the bound pattern is irrefutable, or

  • 5.1.1:8 It does not have a bound pattern and its binding resolves to an irrefutable constant.

  • 5.1.1:9 It does not have a bound pattern and its binding does not resolve to a constant.

5.1.1:10 If the identifier pattern does not have a bound pattern, then the type of its binding is determined as follows:

  • 5.1.1:11 If the identifier pattern has only keyword ref, then the type is & inferred_type, where inferred_type is the type determined by type inference.

  • 5.1.1:12 If the identifier pattern has keywords ref mut, then the type is &mut inferred_type, where inferred_type is the type determined by type inference.

  • 5.1.1:13 Otherwise the type is inferred_type, where inferred_type is the type determined by type inference.

5.1.1:14 If the identifier pattern has a bound pattern, then the type of its binding is determined as follows:

  • 5.1.1:15 If the identifier pattern has only keyword ref, then the type & bound_pattern_type, where bound_pattern_type is the type of the bound pattern.

  • 5.1.1:16 If the identifier pattern has keywords ref mut, then the type is &mut bound_pattern_type, where bound_pattern_type is the type of the bound pattern.

  • 5.1.1:17 Otherwise the type is inferred_type, where inferred_type is the type determined by type inference.

Examples

5.1.1:18 An identifier pattern in the context of a let expression.

let x = 42;

5.1.1:19 An identifier pattern with a bound pattern in the context of a match expression.

match x {
    small @ 1 ..= 5 => (),
    _ => (),
}

5.1.2. Literal Patterns¶

Syntax

LiteralPattern ::=
    BooleanLiteral
  | ByteLiteral
  | ByteStringLiteral
  | CharacterLiteral
  | -? NumericLiteral
  | RawByteStringLiteral
  | RawStringLiteral
  | SimpleStringLiteral

Legality Rules

5.1.2:1 A literal pattern is a pattern that matches a literal.

5.1.2:2 A literal pattern is always refutable.

5.1.2:3 The type of a literal pattern is the type of the specified literal.

Examples

5.1.2:4 Two literal patterns in the context of a match expression. See Paragraph 5.1.1. for the declaration of x.

match x {
    -2 => (),
    36 => (),
    _  => (),
}

5.1.3. Parenthesized Patterns¶

Syntax

ParenthesizedPattern ::=
    ( Pattern )

Legality Rules

5.1.3:1 A parenthesized pattern is a pattern that controls the precedence of its subpatterns.

5.1.3:2 A parenthesized pattern is irrefutable when its nested pattern is irrefutable.

5.1.3:3 The type of a parenthesized pattern is the type of its nested pattern.

Examples

5.1.3:4 See Paragraph 5.1.1. for the declaration of x.

let ref_x = &x;

5.1.3:5 A parenthesized pattern inside a reference pattern in the context of a match expression.

match ref_x {
    &(1 ..= 5) => (),
    _ => (),
}

5.1.4. Path Patterns¶

Syntax

PathPattern ::=
    PathInExpression
  | QualifiedPathInExpression

Legality Rules

5.1.4:1 A path pattern is a pattern that matches a constant, an enum variant without fields, or a unit struct indicated by a path.

5.1.4:2 A path pattern expressed as a path-in-expression shall refer to either an associated constant, or a constant.

5.1.4:3 When a path pattern expressed as a path-in-expression refers to a constant, the constant shall not be of a union type. If the constant is of an enum type or struct type, then the constant shall be subject to attribute derive with arguments core::cmp::Eq and core::cmp::PartialEq.

5.1.4:4 A path pattern expressed as a qualified path-in-expression shall refer to an associated constant.

5.1.4:5 A path pattern is irrefutable when it refers to:

  • 5.1.4:6 A constant whose type is irrefutable, or

  • 5.1.4:7 An enum variant of an enum type with a single enum variant where the types of all fields are irrefutable, or

  • 5.1.4:8 A struct where the types of all fields are irrefutable.

5.1.4:9 The type of a path pattern is the type of the constant, enum, or struct the path resolved to.

Examples

mod module {
     pub const ZERO: i32 = 0;
}

enum Enum { Variant }

5.1.4:10 See Paragraph 5.1.1. for the declaration of x.

match x {
    module::ZERO => (),
    Enum::Variant => (),
    _  => (),
}

5.1.5. Range Patterns¶

Syntax

RangePattern ::=
    HalfOpenRangePattern
  | InclusiveRangePattern
  | ObsoleteRangePattern

HalfOpenRangePattern ::=
    RangePatternLowBound ..

InclusiveRangePattern ::=
    RangePatternLowBound ..= RangePatternHighBound

ObsoleteRangePattern ::=
    RangePatternLowBound ... RangePatternHighBound

RangePatternLowBound ::=
    RangePatternBound

RangePatternHighBound ::=
    RangePatternBound

RangePatternBound ::=
    ByteLiteral
  | CharacterLiteral
  | -? NumericLiteral
  | PathInExpression
  | QualifiedPathInExpression

Legality Rules

5.1.5:1 A range pattern is a pattern that matches values which fall within a range.

5.1.5:2 A half-open range pattern is a range pattern with only a range pattern low bound.

5.1.5:3 An inclusive range pattern is a range pattern with both a range pattern low bound and a range pattern high bound.

5.1.5:4 An obsolete range pattern is a range pattern that uses obsolete syntax to express an inclusive range pattern.

5.1.5:5 A range pattern bound is a constraint on the range of a range pattern.

5.1.5:6 A range pattern low bound is a range pattern bound that specifies the start of a range.

5.1.5:7 A range pattern high bound is a range pattern bound that specifies the end of a range.

5.1.5:8 A half-open range pattern shall appear within a parenthesized pattern when context is a slice pattern.

5.1.5:9 The range pattern low bound of an inclusive range pattern shall be less than or equal to its range pattern high bound.

5.1.5:10 An obsolete range pattern is equivalent to an inclusive range pattern.

5.1.5:11 A range pattern is irrefutable only when it spans the entire set of possible values of a type.

5.1.5:12 The types of the range pattern low bound and the range pattern high bound of a range pattern shall be unifiable.

5.1.5:13 The type of a range pattern is determined as follows:

  • 5.1.5:14 If the range pattern is expressed as an inclusive range pattern or an obsolete range pattern, then the type is the unified type of the types of the range pattern low bound and the range pattern high bound.

  • 5.1.5:15 Otherwise the type is the type of the range pattern low bound.

5.1.5:16 A path-in-expression of a range pattern shall refer to a constant of a scalar type.

5.1.5:17 A qualified path-in-expression of a range pattern shall refer to an associated constant of a scalar type.

Examples

5.1.5:18 Two range patterns in the context of a match expression. See Paragraph 5.1.1. for the declaration of x.

match x {
    -30 ..= 2 => (),
    57 .. => (),
    _ => (),
}

5.1.6. Reference Patterns¶

Syntax

ReferencePattern ::=
    & mut? PatternWithoutRange

Legality Rules

5.1.6:1 A reference pattern is a pattern that dereferences a pointer that is being matched.

5.1.6:2 A reference pattern is always irrefutable.

5.1.6:3 The type of a reference pattern is determined as follows:

  • 5.1.6:4 If the reference pattern appears with keyword mut, then the type is &mut pattern_without_range_type, where pattern_without_range_type is the type of the PatternWithoutRange.

  • 5.1.6:5 If the reference pattern appears without keyword mut, then the type is & pattern_without_range_type, where pattern_without_range_type is the type of the PatternWithoutRange.

Examples

5.1.6:6 A reference pattern in the context of a match expression. See Paragraph 5.1.3. for the declaration of ref_x.

match ref_x {
    &23 => (),
    _ => (),
}

5.1.7. Rest Patterns¶

Syntax

RestPattern ::=
    ..

Legality Rules

5.1.7:1 A rest pattern is a pattern that matches zero or more elements that have not already been matched.

5.1.7:2 A rest pattern shall appear at most once within a slice pattern, an identifier pattern of a slice pattern, a tuple pattern, and a tuple struct pattern.

5.1.7:3 A rest pattern is always irrefutable.

5.1.7:4 If a rest pattern appears within a slice pattern or the identifier pattern of a slice pattern, then the type of the rest pattern is determined as follows:

  • 5.1.7:5 If the type of the slice pattern is an array type, then the type is [T; N] where T is the element type of the array type, and N is the array type's size minus the number of matched elements of the slice pattern.

  • 5.1.7:6 If the type of the slice pattern is a slice type, then the type is that slice type.

Examples

5.1.7:7 A rest pattern in an identifier pattern of a slice pattern, followed by a rest pattern in a slice pattern.

match slice {
    [1, 5, .., 7] => (),
    [start, end @ ..] => (),
}

5.1.7:8 Rest patterns in tuple patterns.

match tuple {
    (1, .., y) => (),
    (.., 5) => (),
    (..) => (),
}

5.1.8. Slice Patterns¶

Syntax

SlicePattern ::=
    [ PatternList? ]

Legality Rules

5.1.8:1 A slice pattern is a pattern that matches arrays of fixed size and slices of dynamic size.

5.1.8:2 A slice pattern is irrefutable when it refers to:

  • 5.1.8:3 An array, where each subpattern is irrefutable, or

  • 5.1.8:4 A slice, where the PatternList consists of a single rest pattern, or a single possibly nested identifier pattern whose last bound pattern is a rest pattern.

5.1.8:5 The type of a slice pattern is the same as the expected type.

Examples

let v = vec![1, 2, 3];

5.1.8:6 A slice pattern in the context of a match expression.

match v {
    [a, b, c] => (),
    _ => ()
}

5.1.9. Struct Patterns¶

Syntax

StructPattern ::=
    RecordStructPattern
  | TupleStructPattern
  | UnitStructPattern

RecordStructPattern ::=
    DeconstructionType { RecordStructDeconstructor? }

RecordStructDeconstructor ::=
    RecordStructRestPattern
  | RecordStructFieldDeconstructorList
  | RecordStructIndexedFieldDeconstructorList

RecordStructRestPattern ::=
    OuterAttributeOrDoc* RestPattern

RecordStructFieldDeconstructorList ::=
    RecordStructFieldDeconstructor (, RecordStructFieldDeconstructor)* (, RecordStructRestPattern | ,?)

RecordStructFieldDeconstructor ::=
    OuterAttributeOrDoc* (
        NamedDeconstructor
      | ShorthandDeconstructor
    )

NamedDeconstructor ::=
    Identifier : Pattern

ShorthandDeconstructor ::=
    ref? mut? Binding

RecordStructIndexedFieldDeconstructorList ::=
    IndexedDeconstructor (, IndexedDeconstructor)* (, RecordStructRestPattern | ,?)

IndexedDeconstructor ::=
    OuterAttributeOrDoc* TupleIndex : Pattern
TupleStructPattern ::=
    DeconstructionType ( PatternList? )
UnitStructPattern ::=
    DeconstructionType
DeconstructionType ::=
    PathInExpression

Legality Rules

5.1.9:1 A struct pattern is a pattern that matches a struct.

5.1.9:2 A tuple struct pattern is a struct pattern that matches enum variants and tuple structs.

5.1.9:3 A union pattern is a record struct pattern that matches a union.

5.1.9:4 A unit struct pattern is a struct pattern that matches a unit struct.

5.1.9:5 An indexed deconstructor is a construct that matches the position of a field of a tuple.

5.1.9:6 A named deconstructor is a construct that matches the name of a field of a struct.

5.1.9:7 A shorthand deconstructor is a construct that matches the name of a field of a struct.

5.1.9:8 The deconstruction type indicates the type of the struct being deconstructed by a struct pattern.

5.1.9:9 A RecordStructPattern without a RecordStructIndexedFieldDeconstructorList is a record struct pattern.

5.1.9:10 A TupleStructPattern and a RecordStructPattern without a RecordStructFieldDeconstructorList are tuple struct patterns.

5.1.9:11 A RecordStructPattern with a RecordStructFieldDeconstructorList is a union pattern.

5.1.9:12 A UnitStructPattern and a RecordStructPattern without a RecordStructFieldDeconstructorList and a RecordStructIndexedFieldDeconstructorList are unit struct patterns.

5.1.9:13 A struct pattern is interpreted based on the deconstruction type. It is a static error if a struct pattern cannot be interpreted.

5.1.9:14 A shorthand deconstructor binds the value of a matched field to a variable. A shorthand deconstructor with keyword mut yields a mutable variable.

5.1.9:15 It is a static error if a shorthand deconstructor has only keyword ref or keywords ref mut, and its variable shadows a constant.

5.1.9:16 A struct pattern is irrefutable when all of its subpatterns are irrefutable.

5.1.9.1. Record Struct Patterns¶

Legality Rules

5.1.9.1:1 The deconstruction type of a record struct pattern shall resolve to a record struct type.

5.1.9.1:2 A named deconstructor matches a field of the deconstruction type when its identifier and the name of the field are the same. Such a named deconstructor is a matched named deconstructor.

5.1.9.1:3 The type of a named deconstructor and the type of a matched field shall be unifiable.

5.1.9.1:4 A shorthand deconstructor matches a field of the deconstruction type when its name and the name of the field are the same. Such a shorthand deconstructor is a matched shorthand deconstructor.

5.1.9.1:5 The type of a shorthand deconstructor and the type of the matched field shall be unifiable.

5.1.9.1:6 A shorthand deconstructor is equivalent to a named deconstructor where the name of the shorthand deconstructor denotes the identifier of the named deconstructor and the entire content of the shorthand deconstructor denotes the pattern of the named deconstructor.

5.1.9.1:7 For each field of the deconstruction type, the record struct deconstructor shall either:

  • 5.1.9.1:8 Contain a matched named deconstructor, or

  • 5.1.9.1:9 Contain a matched shorthand deconstructor, or

  • 5.1.9.1:10 Has a RecordStructRestPattern or a RecordStructFieldDeconstructorList with a RecordStructRestPattern.

Examples

struct Struct {
     field: u32,
     other: u32,
}

let Struct { field, other };
let Struct { field, .. };
let Struct { .. };

5.1.9.2. Tuple Struct Patterns¶

Legality Rules

5.1.9.2:1 The deconstruction type of a tuple struct pattern shall resolve to a tuple struct type.

5.1.9.2:2 A tuple struct pattern shall contain one subpattern for each field of the deconstruction type.

5.1.9.2:3 A subpattern of a tuple struct pattern matches a field of the deconstruction type when its position and the position of the field in the deconstruction type are the same. Such a subpattern is a matched subpattern.

5.1.9.2:4 The type of the subpattern of a tuple struct pattern and the type of the matched field shall be unifiable.

5.1.9.2:5 An index deconstructor matches a field of the deconstruction type when its tuple index and the position of the field in the deconstruction type are the same. Such an index deconstructor is a matched index deconstructor.

5.1.9.2:6 For each field of the deconstruction type, the tuple struct deconstructor shall either:

  • 5.1.9.2:7 Contain a matched index deconstructor, or

  • 5.1.9.2:8 Contain a matched subpattern, or

  • 5.1.9.2:9 Has a RecordStructRestPattern.

Examples

struct Tuple(u32, f32);

let Tuple(first, second);
let Tuple(first, ..);
let Tuple(..);

5.1.9.3. Union Patterns¶

Legality Rules

5.1.9.3:1 The deconstruction type of a union pattern shall resolve to a union type.

5.1.9.3:2 The RecordStructFieldDeconstructorList of a union pattern shall contain exactly one RecordStructFieldDeconstructor and no RecordStructRestPattern.

5.1.9.3:3 For the single field of the deconstruction type, a union deconstructor shall either:

  • 5.1.9.3:4 Contain a matched named deconstructor, or

  • 5.1.9.3:5 Contain a matched shorthand deconstructor.

5.1.9.3:6 A union pattern shall require unsafe context.

Examples

union Union {
    int: u32,
    float: f32,
}

unsafe {
    let Union { int } = Union { int: 0 };
}

5.1.9.4. Unit Struct Patterns¶

Legality Rules

5.1.9.4:1 The deconstruction type of a unit struct deconstructor shall resolve to a unit struct type.

Examples

struct Empty;

let Empty = Empty;
let Empty = Empty{};

5.1.10. Tuple Patterns¶

Syntax

TuplePattern ::=
    ( TuplePatternElementList? )
TuplePatternElementList ::=
    Pattern ,
  | PatternList
  | RestPattern

Legality Rules

5.1.10:1 A tuple pattern is a pattern that matches a tuple which satisfies all criteria defined by its subpatterns.

5.1.10:2 A tuple pattern is irrefutable when all of its subpatterns are irrefutable.

5.1.10:3 The type of a tuple pattern is the type of the tuple being destructured.

Examples

let pair = (1, "two");

5.1.10:4 A tuple pattern in the context of a let statement.

let (first, second) = pair;

5.1.11. Wildcard Patterns¶

Syntax

WildcardPattern ::=
    _

Legality Rules

5.1.11:1 A wildcard pattern is a pattern that matches any single value.

5.1.11:2 A wildcard pattern is always irrefutable.

5.1.11:3 The type of a wildcard pattern is the type of the value it matches.

Examples

5.1.11:4 A wildcard pattern in the context of a let statement. See Paragraph 5.1.10. for the declaration of pair.

let (first, _) = pair;

5.2. Refutability¶

Legality Rules

5.2:1 Refutability is a property of patterns that expresses the ability to match all possible values of a type.

5.2:2 An irrefutable pattern is a pattern that always matches the value it is being matched against.

5.2:3 A refutable pattern is a pattern that has a possibility of not matching the value it is being matched against.

5.2:4 An irrefutable type is a type that has at most one value.

5.2:5 A refutable type is a type that has more than one value.

5.2:6 An irrefutable constant is a constant of an irrefutable type.

5.2:7 A refutable constant is a constant of a refutable type.

Examples

5.2:8 x is an irrefutable pattern because it always matches 42.

let x = 42;

5.2:9 y is a refutable pattern because it does not match value when value denotes core::option::Option::None.

if let core::option::Option::Some(y) = value {

5.3. Binding Modes¶

Syntax

Binding ::=
    Name

Legality Rules

5.3:1 A binding pattern is either an identifier pattern or a shorthand deconstructor.

5.3:2 A binding is a variable of a binding pattern that binds a matched value.

5.3:3 A non-reference pattern is any pattern except non-binding patterns, path patterns, reference patterns, and wildcard patterns.

5.3:4 If a binding pattern does not explicitly specify keyword ref, keyword mut, or keywords ref mut, then its binding mode uses the current binding mode of pattern matching.

5.3:5 Initially, the binding mode of a binding is by value.

5.3:6 During the process of pattern matching, each time a reference is matched against a non-reference pattern, the reference is dereferenced and the binding mode is updated as follows:

  • 5.3:7 If the reference is an immutable reference, then the binding mode is updated to by reference.

  • 5.3:8 If the reference is a mutable reference and the binding mode is “by value”, then the binding mode is updated to “by mutable reference”.

5.3:9 The process repeats if the dereferenced value is a reference.

Dynamic Semantics

5.3:10 A binding patterns binds its binding to a matched value as follows:

  • 5.3:11 If the binding mode is by reference or the binding pattern appears only with keyword ref, then the binding is bound to a reference of the matched value.

  • 5.3:12 If the binding mode is by mutable reference or the binding pattern appears with keywords ref mut, then the binding is bound to a mutable reference of the matched value.

  • 5.3:13 If the binding mode is by value, then the binding is bound to a copy of the matched value if its type implements the core::marker::Copy trait, otherwise the binding is bound to the move of the matched value.

5.4. Pattern Matching¶

Dynamic Semantics

5.4:1 Pattern matching that involves a pattern and a context value proceeds as follows:

  1. 5.4:2 For each pattern-without-alternation of the pattern

    1. 5.4:3 If the pattern-without-alternation is an identifier pattern, then perform identifier pattern matching.

    2. 5.4:4 If the pattern-without-alternation is a literal pattern, then perform literal pattern matching.

    3. 5.4:5 If the pattern-without-alternation is a parenthesized pattern, then perform parenthesized pattern matching.

    4. 5.4:6 If the pattern-without-alternation is a path pattern, then perform path pattern matching.

    5. 5.4:7 If the pattern-without-alternation is a range pattern, then perform range pattern matching.

    6. 5.4:8 If the pattern-without-alternation is a reference pattern, then perform reference pattern matching.

    7. 5.4:9 If the pattern-without-alternation is a slice pattern, then perform slice pattern matching.

    8. 5.4:10 If the pattern-without-alternation is a struct pattern, then perform struct pattern matching.

    9. 5.4:11 If the pattern-without-alternation is a tuple pattern, then perform tuple pattern matching.

    10. 5.4:12 If the pattern-without-alternation is a tuple struct pattern, then perform tuple struct pattern matching.

    11. 5.4:13 If the pattern-without-alternation is a wildcard pattern, then perform wildcard pattern matching.

    12. 5.4:14 Otherwise pattern matching fails.

5.4:15 Only the bindings of a matched pattern-without-alternation are introduced to the corresponding scope.

5.4.1. Identifier Pattern Matching¶

Legality Rules

5.4.1:1 An identifier pattern with keyword mut shall require that the context value is a mutable place expression.

Dynamic Semantics

5.4.1:2 Identifier pattern matching proceeds as follows:

  1. 5.4.1:3 If the identifier pattern has a bound pattern, then

    1. 5.4.1:4 Performed pattern matching with the bound pattern and the same context value.

    2. 5.4.1:5 If matching the bound pattern fails, then matching fails.

  2. 5.4.1:6 The context value is bound to the binding of the identifier pattern according to the binding mode.

  3. 5.4.1:7 Matching succeeds.

5.4.2. Literal Pattern Matching¶

Dynamic Semantics

5.4.2:1 Literal pattern matching proceeds as follows:

  1. 5.4.2:2 If the literal of the literal pattern and the context value are equal, then matching succeeds.

  2. 5.4.2:3 Otherwise matching fails.

5.4.3. Parenthesized Pattern Matching¶

Dynamic Semantics

5.4.3:1 Parenthesized pattern matching performs pattern matching with its subpattern and the same context value.

5.4.4. Path Pattern Matching¶

Dynamic Semantics

5.4.4:1 Path pattern matching proceeds as follows:

  1. 5.4.4:2 If the constant the path of the path pattern resolved to and the context value are equal, then matching succeeds.

  2. 5.4.4:3 Otherwise matching fails.

5.4.5. Range Pattern Matching¶

Dynamic Semantics

5.4.5:1 Range pattern matching proceeds as follows:

  1. 5.4.5:2 If the range pattern is expressed as a half-open range pattern and the context value is in the inclusive range from the range pattern low bound to the maximum value of the range pattern low bound's type, then matching succeeds.

  2. 5.4.5:3 If the range pattern is expressed as either an inclusive range pattern or an obsolete range pattern and the context value is in the inclusive range from the range pattern low bound to the range pattern high bound, then matching succeeds.

  3. 5.4.5:4 Otherwise matching fails.

5.4.6. Reference Pattern Matching¶

Dynamic Semantics

5.4.6:1 Reference pattern matching proceeds as follows:

  1. 5.4.6:2 Dereference the context value.

  2. 5.4.6:3 Perform pattern matching with its subpattern and the dereferenced value.

5.4.7. Slice Pattern Matching¶

Dynamic Semantics

5.4.7:1 Slice pattern matching proceeds as follows if the expected type is an array type or a slice type:

  1. 5.4.7:2 If the number of subpatterns of the slice pattern is greater than the length of the context value, then matching fails.

  2. 5.4.7:3 If the number of subpatterns of the slice pattern is less than the size of the context value and one of those subpatterns is not a rest pattern, then matching fails.

  3. 5.4.7:4 For each subpattern of the slice pattern

    1. 5.4.7:5 Perform pattern matching with the subpattern and the corresponding value from the context value, ignoring rest patterns.

    2. 5.4.7:6 If matching the subpattern fails, then matching fails.

5.4.8. Struct Pattern Matching¶

Dynamic Semantics

5.4.8:1 Struct pattern matching proceeds as follows:

  1. 5.4.8:2 If the number of subpatterns of the struct pattern is greater than the number of fields of the context value, then this is a static error.

  2. 5.4.8:3 If the number of subpatterns of the struct pattern is less than the number of fields of the context value and one of those subpatterns is not a rest pattern, then matching fails.

  3. 5.4.8:4 For each subpattern of the struct pattern

    1. 5.4.8:5 If the subpattern is a shorthand deconstructor, then the corresponding field of the context value is bound to the binding of the shorthand deconstructor according to the binding mode.

    2. 5.4.8:6 Otherwise perform pattern matching with the subpattern and the corresponding field from the context value, ignoring rest patterns.

    3. 5.4.8:7 If matching the subpattern fails, then matching fails.

5.4.9. Tuple Pattern Matching¶

Dynamic Semantics

5.4.9:1 Tuple pattern matching proceeds as follows:

  1. 5.4.9:2 If the number of subpatterns of the tuple pattern is greater than the number of fields of the context value, then this is a static error.

  2. 5.4.9:3 If the number of subpatterns of the tuple pattern is less than the number of fields of the context value and one of those subpatterns is not a rest pattern, then this is a static error.

  3. 5.4.9:4 For each subpattern of the tuple pattern

    1. 5.4.9:5 Perform pattern matching with the subpattern and the corresponding field from the context value, ignoring rest patterns.

    2. 5.4.9:6 If matching the subpattern fails, then matching fails.

5.4.10. Tuple Struct Pattern Matching¶

Dynamic Semantics

5.4.10:1 Tuple struct pattern matching proceeds as follows:

  1. 5.4.10:2 If the number of subpatterns of the tuple struct pattern is greater than the number of fields of the context value, then this is a static error.

  2. 5.4.10:3 If the number of subpatterns of the tuple struct pattern is less than the number of fields of the context value and one of those subpatterns is not a rest pattern, then this is a static error.

  3. 5.4.10:4 For each subpattern of the tuple struct pattern

    1. 5.4.10:5 Otherwise perform pattern matching with the subpattern and the corresponding field from the context value, ignoring rest patterns.

    2. 5.4.10:6 If matching the subpattern fails, then matching fails.

5.4.11. Wildcard Pattern Matching¶

Dynamic Semantics

5.4.11:1 Wildcard pattern matching proceeds as follows:

  1. 5.4.11:2 The context value is matched unconditionally.

  2. 5.4.11:3 Matching succeeds.
