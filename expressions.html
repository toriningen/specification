6. Expressions¶

Syntax

Expression ::=
    ExpressionWithBlock
  | ExpressionWithoutBlock

ExpressionWithBlock ::=
    OuterAttributeOrDoc* (
        AsyncBlockExpression
      | BlockExpression
      | IfExpression
      | IfLetExpression
      | LoopExpression
      | MatchExpression
      | UnsafeBlockExpression
    )

ExpressionWithoutBlock ::=
    OuterAttributeOrDoc* (
        ArrayExpression
      | ArrayIndexExpression
      | AwaitExpression
      | BreakExpression
      | CallExpression
      | ClosureExpression
      | ContinueExpression
      | FieldAccessExpression
      | LiteralExpression
      | MethodCallExpression
      | MacroInvocation
      | OperatorExpression
      | ParenthesizedExpression
      | PathExpression
      | RangeExpression
      | ReturnExpression
      | StructExpression
      | TupleExpression
      | UnderscoreExpression
    )

ExpressionList ::=
    Expression (, Expression)* ,?

Operand ::=
    Expression

LeftOperand ::=
    Operand

RightOperand ::=
    Operand

6:1 A SubjectExpression is any expression in category Expression, except IndexedTupleStructExpression and RecordStructExpression.

6:2 A SubjectLetExpression is any expression in category SubjectExpression, except LazyBooleanExpression.

Legality Rules

6:3 An expression is a construct that produces a value, and may have side effects at run-time.

6:4 An expression-with-block is an expression whose structure involves a block expression.

6:5 An expression-without-block is an expression whose structure does not involve a block expression.

6:6 An operand is an expression nested within an expression.

6:7 A left operand is an operand that appears on the left-hand side of a binary operator.

6:8 A right operand is an operand that appears on the right-hand side of a binary operator.

6:9 A subject expression is an expression that controls for loops, if expressions, and match expressions.

6:10 A subject let expression is an expression that controls if let expressions and while let loops.

Dynamic Semantics

6:11 Evaluation is the process by which an expression achieves its runtime effects.

6.1. Expression Classification¶

6.1.1. Assignee Expressions¶

Legality Rules

6.1.1:1 An assignee expression is an expression that appears as the left operand of an assignment expression. The following expressions are assignee expressions:

  • 6.1.1:2 Place expressions,

  • 6.1.1:3 Underscore expressions,

  • 6.1.1:4 Array expressions of assignee expressions,

  • 6.1.1:5 Tuple expressions of assignee expressions,

  • 6.1.1:6 Struct expressions of assignee expressions,

  • 6.1.1:7 Tuple struct expressions of assignee expressions,

  • 6.1.1:8 Unit struct expressions.

6.1.1:9 Parenthesized expressions are allowed to appear anywhere in assignee expressions.

6.1.2. Constant Expressions¶

Legality Rules

6.1.2:1 A constant expression is an expression that can be evaluated statically. The following constructs are constant expressions as long as their operands are also constant expressions and do not involve types that require destruction:

  • 6.1.2:2 Arithmetic expressions of scalar types,

  • 6.1.2:3 Array expressions,

  • 6.1.2:4 Array index expressions,

  • 6.1.2:5 Assignment expressions,

  • 6.1.2:6 Bit expressions of scalar types,

  • 6.1.2:7 Block expressions,

  • 6.1.2:8 Call expressions where the callee is a constant function,

  • 6.1.2:9 Closure expressions that do not capture,

  • 6.1.2:10 Comparison expressions of scalar types,

  • 6.1.2:11 Compound assignment expressions,

  • 6.1.2:12 Constant parameters,

  • 6.1.2:13 Dereference expressions when the operand is not of a raw pointer type,

  • 6.1.2:14 Expression statements,

  • 6.1.2:15 Field access expressions that do not invoke the core::ops::Deref trait,

  • 6.1.2:16 If expressions,

  • 6.1.2:17 If let expressions,

  • 6.1.2:18 Immutable borrow expressions where the borrowed type is not subject to interior mutability.

  • 6.1.2:19 Infinite loop expressions,

  • 6.1.2:20 Lazy boolean expressions of scalar types,

  • 6.1.2:21 Let statements,

  • 6.1.2:22 Literal expressions,

  • 6.1.2:23 Match expressions,

  • 6.1.2:24 Method call expressions where the callee is a constant function or do not invoke the core::ops::Deref trait,

  • 6.1.2:25 Negation expressions of scalar types,

  • 6.1.2:26 Parenthesized expressions,

  • 6.1.2:27 Path expressions that resolve to associated constants, constants, constant generic parameters, functions, statics, tuple structs, and unit structs,

  • 6.1.2:28 Range expressions,

  • 6.1.2:29 Shared borrows that do not involve types with interior mutability,

  • 6.1.2:30 Struct expressions,

  • 6.1.2:31 Tuple expressions,

  • 6.1.2:32 Type cast expressions that are not pointer-to-address casts, function-pointer-to-address casts, and unsized casts that involve a trait object type,

  • 6.1.2:33 Unsafe block expressions,

  • 6.1.2:34 While let loop expressions,

  • 6.1.2:35 While loop expressions.

6.1.2:36 An expression is not considered a constant expression when it explicitly invokes an associated trait function or uses arithmetic operators of non-builtin types that invoke core::ops traits.

6.1.2:37 An arithmetic expression that operates with floating point values is not considered a constant expression when it appears inside the body of a constant function.

6.1.2:38 It is a static error if the size operand of an array repetition constructor or an array type depends on generic parameters.

6.1.2:39 A constant context is a construct that requires a constant expression. The following constructs are constant contexts:

  • 6.1.2:40 The constant initializer of an associated constant or a constant,

  • 6.1.2:41 The constant argument for a constant generic parameter,

  • 6.1.2:42 The default value of a constant generic parameter,

  • 6.1.2:43 The discriminant initializer of an enum variant,

  • 6.1.2:44 The size operand of an array repetition constructor,

  • 6.1.2:45 The size operand of an array type,

  • 6.1.2:46 The static initializer of a static.

6.1.2:47 It is a static error to create a mutable reference in a constant context.

6.1.2:48 An invocation of the addr_of!() macro expands to a constant expression allowed in any constant context and constant function, subject to the same restrictions as a shared borrow expression.

6.1.2:49 An invocation of the panic!() macro expands to a constant expression allowed in any constant context and constant function, as long as the macro is either invoked without arguments, or with a single string literal that does not capture formatting arguments.

6.1.2:50 A constant expression is evaluated statically whenever its value is needed.

6.1.2:51 The evaluation of a constant expression that results in arithmetic overflow panics.

6.1.2:52 It is a static error if a constant expression either panics or control reaches the invocation of macro core::panic.

Dynamic Semantics

6.1.2:53 The invocation of a constant function follows the dynamic semantics of a non-constant function invocation.

6.1.3. Place Expressions¶

Legality Rules

6.1.3:1 A place expression is an expression that represents a memory location. The following expressions are place expressions:

  • 6.1.3:2 Array index expressions,

  • 6.1.3:3 Dereference expressions,

  • 6.1.3:4 Field access expressions,

  • 6.1.3:5 Parenthesized expressions where the operand is a place expression,

  • 6.1.3:6 Path expressions that resolve to a binding or a static.

6.1.3:7 A place expression context is a construct that requires a place expression. The following constructs are place expression contexts:

  • 6.1.3:8 The indexed array operand of an array index expression,

  • 6.1.3:9 The assignee operand of an assignment expression or a compound assignment expression,

  • 6.1.3:10 The operand of a borrow expression,

  • 6.1.3:11 The operand of a dereference expression,

  • 6.1.3:12 The container operand of field access expression,

  • 6.1.3:13 The subject let expression of an if let expression or a while let loop expression,

  • 6.1.3:14 The initialization expression of a let statement,

  • 6.1.3:15 The subject expression of a match expression,

  • 6.1.3:16 The base initializer of a struct expression,

  • 6.1.3:17 The operand of an implicit borrow.

6.1.3:18 A place expression can be moved out of when it denotes

  • 6.1.3:19 A binding which is not currently borrowed, or

  • 6.1.3:20 A field of a place expression that can be moved out of and does not implement the core::ops::Drop trait, or

  • 6.1.3:21 A temporary created for a value expression.

6.1.3:22 After a place expression is moved out, the memory location it represented is deinitialized and shall not be read from until reinitialized.

6.1.3:23 A mutable place expression is a place expression whose memory location can be modified. The following place expressions are mutable place expressions:

  • 6.1.3:24 An array index expression whose type implements the core::ops::IndexMut trait,

  • 6.1.3:25 A dereference expression whose type is *mut T,

  • 6.1.3:26 A dereference expression of a field or binding whose type is &mut T,

  • 6.1.3:27 A dereference expression whose type implements the core::ops::DerefMut trait,

  • 6.1.3:28 A field access expression where the type of the container operand is mutable,

  • 6.1.3:29 A path expression that resolves to a mutable binding that is not currently borrowed,

  • 6.1.3:30 A path expression that resolves to a mutable static,

  • 6.1.3:31 A temporary created for a value expression.

Dynamic Semantics

6.1.3:32 The evaluation of a place expression in the context of a value expression or the evaluation of a place expression that is bound by value in a pattern proceeds as follows:

  1. 6.1.3:33 The place expression denotes the value held in that memory location.

  2. 6.1.3:34 If the type of the held value implements the core::marker::Copy trait, then the held value is copied.

  3. 6.1.3:35 If the type of the held value implements the core::marker::Sized trait, then the held value is moved.

  4. 6.1.3:36 Otherwise the evaluation results in a static error.

Syntax

PathExpression ::=

Legality Rules

6.3:1 A path expression is an expression that denotes a path.

6.3:2 A path expression that resolves to a binding or a static is a place expression, otherwise it is a value expression.

6.3:3 A path expression that resolves to a mutable static shall require unsafe context.

6.3:4 The type of a path expression is the type of the entity that it resolved to.

Dynamic Semantics

6.3:6 The evaluation of a path expression has no effect.

6.4. Block Expressions¶

Syntax

BlockExpression ::=
    {
      InnerAttributeOrDoc*
      StatementList
    }
StatementList ::=
    Statement* Expression?

Legality Rules

6.4:1 A block expression is an expression that sequences expressions and statements.

6.4:2 A tail expression is the last expression within a block expression.

6.4:3 A block expression is a value expression.

6.4:4 The type of a block expression is determined as follows:

  • 6.4:5 If the block expression has an expression, then the type is the type of the expression.

  • 6.4:6 If the block expression does not have an expression, then the type is the unit type.

6.4:7 The value of a block expression is determined as follows:

  • 6.4:8 If the block expression has an expression, then the value is the value of the expression.

  • 6.4:9 If the block expression does not have an expression, then the value of the block expression is the unit value.

Dynamic Semantics

6.4:10 The evaluation of a block expression proceeds as follows:

  1. 6.4:11 Each statement is executed in declarative order.

  2. 6.4:12 The expression is evaluated.

Examples

{
    fn_call();
    42
}

6.4.1. async Blocks¶

Syntax

AsyncBlockExpression ::=
    async move? BlockExpression

Legality Rules

6.4.1:1 An async block expression is a block expression that ???.

6.4.1:2 An async block expression is a value expression.

6.4.1:3 An async block expression is subject to capturing.

6.4.1:4 An async block expression denotes a new control flow boundary.

6.4.1:5 The type of an async block expression shall implement the core::future::Future trait.

6.4.1:6 The value of an async block expression is a future.

Dynamic Semantics

6.4.1:7 The evaluation of an async block expression produces an anonymous object that captures the related future.

Examples

async {
    42
}

6.4.2:3 An unsafe block expression is a value expression.

6.4.2:4 The type of the unsafe block expression is the type of its block expression.

6.4.2:5 The value of the unsafe block expression is the value of its block expression.

6.4.2:6 The evaluation of an unsafe block expression evaluates its block expression.

6.5. Operator Expressions¶

Syntax

OperatorExpression ::=
    ArithmeticExpression
  | AssignmentExpression
  | BitExpression
  | BorrowExpression
  | ComparisonExpression
  | CompoundAssignmentExpression
  | DereferenceExpression
  | ErrorPropagationExpression
  | LazyBooleanExpression

Legality Rules

6.5:1 An operator expression is an expression that involves an operator.

6.5.1. Borrow Expression¶

Syntax

BorrowExpression ::=
    & mut? Operand

Legality Rules

6.5.1:1 A borrow expression is an expression that borrows the value of its operand and creates a reference to the memory location of its operand.

6.5.1:2 An immutable borrow expression is a borrow expression that lacks keyword mut.

6.5.1:3 A mutable borrow expression is a borrow expression that has keyword mut.

6.5.1:4 When the operand of a borrow expression is a place expression, the borrow expression produces a reference to the memory location indicated by the operand. The memory location is placed in a borrowed state, or simply borrowed.

6.5.1:5 When the operand of a borrow expression is a value expression, a temporary is allocated and the borrow expression produces a reference to the memory location of the temporary.

6.5.1:6 A borrow expression is a value expression.

6.5.1:7 The type of a borrow expression is determined as follows:

  • 6.5.1:8 If the borrow expression denotes a shared reference, then the type is &T where T is the type of the operand.

  • 6.5.1:9 If the borrow expression denotes a mutable reference, then the type is &mut T where T is the type of the operand.

6.5.1:10 The value of a borrow expression is the address of its operand.

Dynamic Semantics

6.5.1:11 The evaluation of a borrow expression evaluates its operand.

Examples

let mut answer = 42;

6.5.1:12 Mutable borrow.

let ref_answer = &mut answer;

6.5.2. Dereference Expression¶

Syntax

DereferenceExpression ::=
    * Operand

Legality Rules

6.5.2:1 A dereference expression is an expression that obtains the pointed-to memory location of its operand.

6.5.2:2 When the operand of a dereference expression is of a pointer type, the dereference expression denotes the pointed-to memory location of the operand, or the dereference of the operand.

6.5.2:3 The dereference is assignable when

  • 6.5.2:4 The operand is of type &mut T or *mut T, and

  • 6.5.2:5 The operand is a binding or a possibly nested field of a binding, or

  • 6.5.2:6 The operand denotes a mutable place expression.

  • 6.5.2:7 The operand is of another type that implements the core::ops::DerefMut trait.

6.5.2:8 Dereferencing a raw pointer shall require unsafe context.

6.5.2:9 If the context of a dereference expression is an immutable place expression, then the dereference expression is equivalent to expression *core::ops::Deref::deref(&operand).

6.5.2:10 If the context of a dereference expression is a mutable place expression, then the dereference expression is equivalent to expression *core::ops::DerefMut::deref_mut(&mut operand).

6.5.2:11 The type of a dereference expression is determined as follows:

  • 6.5.2:12 If the type of the operand is &mut T, &T, *mut T, or *const T, then the type is T.

  • 6.5.2:13 Otherwise the type is associated type core::ops::Deref::Target.

6.5.2:14 The value of a dereference expression is determined as follows:

  • 6.5.2:15 If the type of the operand is &mut T, &T, *mut T, or *const T, then the value is the pointed-to value.

  • 6.5.2:16 Otherwise the value is the result of evaluating expression *core::ops::Deref::deref(&operand) or expression *core::ops::DerefMut::deref_mut(&mut operand) respectively.

Dynamic Semantics

6.5.2:17 The evaluation of a dereference expression evaluates its operand.

Undefined Behavior

6.5.2:18 It is undefined behavior to dereference a raw pointer that is either dangling or unaligned.

Examples

6.5.2:19 See Paragraph 6.4.1. for the declaration of ref_answer.

let deref_asnwer = *ref_answer;

6.5.3. Error Propagation Expression¶

Syntax

ErrorPropagationExpression ::=
    Operand ErrorPropagationOperator

ErrorPropagationOperator ::=
    ?

Legality Rules

6.5.3:1 An error propagation expression is an expression that either evaluates to a value of its operand or returns a value to the next control flow boundary.

6.5.3:2 An error propagation expression shall appear within a control flow boundary.

6.5.3:3 The type of an error propagation expression is associated type core::ops::Try::Output.

6.5.3:4 The value of an error propagation expression is determined as follows:

  • 6.5.3:5 If the evaluation of the error propagation expression executed core::ops::Try::branch(operand), then the value is the value of the core::ops::ControlFlow::Continue variant.

  • 6.5.3:6 Otherwise control flow is returned to the end of the enclosing control flow boundary.

6.5.3:7 The expression context for the operand of the error propagation expression is a value expression context.

Dynamic Semantics

6.5.3:8 The evaluation of an error propagation operator of the form

expression?

6.5.3:9 is equivalent to the evaluation the following expression:

match core::ops::Try::branch(expression) {
    core::ops::ControlFlow::Continue(value) =>
        value,

    core::ops::ControlFlow::Break(value) =>
        core::ops::FromResidual::from_residual(value),
}

Examples

fn try_to_parse() -> Result<i32, ParseIntError> {
    "42".parse()?
}

fn try_some() -> Option<i32> {
    let val = Some(42)?;
    Some(val)
}

6.5.4. Negation Expression¶

Syntax

NegationExpression ::=
    NegationOperator Operand

NegationOperator ::=
    BitwiseNegationOperator
  | SignNegationOperator

BitwiseNegationOperator ::=
    !

SignNegationOperator ::=
    -

Legality Rules

6.5.4:1 A negation expression is an expression that negates its operand.

6.5.4:2 The type of the operand of a negation expression with a BitwiseNegationOperator shall implement the core::ops::Not trait.

6.5.4:3 The type of a negation expression with a BitwiseNegationOperator is associated type core::ops::Not::Output.

6.5.4:4 The value of a negation expression with a BitwiseNegationOperator is the result of core::ops::Not::not(operand).

6.5.4:5 The type of the operand of a negation expression with a SignNegationOperator shall implement the core::ops::Neg trait.

6.5.4:6 The type of a negation expression with a SignNegationOperator shall be associated type core::ops::Neg::Output.

6.5.4:7 The value of a negation expression with a SignNegationOperator is the result of core::ops::Neg::neg(operand).

6.5.4:8 The expression context for the operand of the negation expression is a value expression context.

Dynamic Semantics

6.5.4:9 The evaluation of a negation expression with a BitwiseNegationOperator proceeds as follows:

  1. 6.5.4:10 The operand is evaluated.

  2. 6.5.4:11 core::ops::Not::not(operand) is invoked.

6.5.4:12 The evaluation of a negation expression with a SignNegationOperator proceeds as follows:

  1. 6.5.4:13 The operand is evaluated.

  2. 6.5.4:14 core::ops::Neg::neg(operand) is invoked.

Examples

6.5.4:15 Sign negation.

-42

6.5.4:16 Bitwise negation.

!42

6.5.4:17 Logical negation.

!false

6.5.5. Arithmetic Expressions¶

Syntax

ArithmeticExpression ::=
    AdditionExpression
  | DivisionExpression
  | MultiplicationExpression
  | RemainderExpression
  | SubtractionExpression

AdditionExpression ::=
    LeftOperand + RightOperand

DivisionExpression ::=
    LeftOperand / RightOperand

MultiplicationExpression ::=
    LeftOperand * RightOperand

RemainderExpression ::=
    LeftOperand % RightOperand

SubtractionExpression ::=
    LeftOperand - RightOperand

Legality Rules

6.5.5:1 An arithmetic expression is an expression that computes a value from two operands using arithmetic.

6.5.5:2 A division expression is an arithmetic expression that uses division.

6.5.5:3 A multiplication expression is an arithmetic expression that uses multiplication.

6.5.5:4 A remainder expression is an arithmetic expression that uses remainder division.

6.5.5:5 A subtraction expression is an arithmetic expression that uses subtraction.

6.5.5:6 The type of the left operand of an addition expression shall implement the core::ops::Add trait with the type of the right operand as the trait implementation type parameter.

6.5.5:7 The type of an addition expression is associated type core::ops::Add::Output.

6.5.5:8 The value of an addition expression is the result of core::ops::Add::add(left_operand, right_operand).

6.5.5:9 The type of the left operand of a division expression shall implement the core::ops::Div trait where the type of the right operand is the trait implementation type parameter.

6.5.5:10 The type of a division expression is associated type core::ops::Div::Output.

6.5.5:11 The value of a division expression is the result of core::ops::Div::div(left_operand, right_operand).

6.5.5:12 The type of the left operand of a multiplication expression shall implement the core::ops::Mul trait where the type of the right operand is the trait implementation type parameter.

6.5.5:13 The type of a multiplication expression is associated type core::ops::Mul::Output.

6.5.5:14 The value of a multiplication expression is the result of core::ops::Mul::mul(left_operand, right_operand).

6.5.5:15 The type of the left operand of a remainder expression shall implement the core::ops::Rem trait where the type of the right operand is the trait implementation type parameter.

6.5.5:16 The type of a remainder expression is associated type core::ops::Rem::Output.

6.5.5:17 The value of a remainder expression is the result of core::ops::Rem::rem(left_operand, right_operand).

6.5.5:18 The type of the left operand of a subtraction expression shall implement the core::ops::Sub trait where the type of the right operand is the trait implementation type parameter.

6.5.5:19 The type of a subtraction expression is associated type core::ops::Sub::Output.

6.5.5:20 The value of a subtraction expression is the result of core::ops::Sub::sub(left_operand, right_operand).

6.5.5:21 The expression context for the operands of an arithmetic expression is a value expression context.

Dynamic Semantics

6.5.5:22 The evaluation of an addition expression proceeds as follows:

  1. 6.5.5:23 The left operand is evaluated.

  2. 6.5.5:24 The right operand is evaluated.

  3. 6.5.5:25 core::ops::Add::add(left_operand, right_operand) is invoked.

6.5.5:26 The evaluation of a division expression proceeds as follows:

  1. 6.5.5:27 The left operand is evaluated.

  2. 6.5.5:28 The right operand is evaluated.

  3. 6.5.5:29 core::ops::Div::div(left_operand, right_operand) is invoked.

6.5.5:30 The evaluation of a multiplication expression proceeds as follows:

  1. 6.5.5:31 The left operand is evaluated.

  2. 6.5.5:32 The right operand is evaluated.

  3. 6.5.5:33 core::ops::Mul::mul(left_operand, right_operand) is invoked.

6.5.5:34 The evaluation of a remainder expression proceeds as follows:

  1. 6.5.5:35 The left operand is evaluated.

  2. 6.5.5:36 The right operand is evaluated.

  3. 6.5.5:37 core::ops::Rem::rem(left_operand, right_operand) is invoked.

6.5.5:38 The evaluation of a subtraction expression proceeds as follows:

  1. 6.5.5:39 The left operand is evaluated.

  2. 6.5.5:40 The right operand is evaluated.

  3. 6.5.5:41 core::ops::Rem::rem(left_operand, right_operand) is invoked.

Undefined Behavior

6.5.5:42 It is undefined behavior for an arithmetic operation to cause overflow with values of numeric types.

Examples

1 + 2
4.0 / 3.29
8.4 * 5.3
10 % 4
3 - 2

6.5.6. Bit Expressions¶

Syntax

BitExpression ::=
    BitAndExpression
  | BitOrExpression
  | BitXOrExpression
  | ShiftLeftExpression
  | ShiftRightExpression

BitAndExpression ::=
    LeftOperand & RightOperand

BitOrExpression ::=
    LeftOperand | RightOperand

BitXorExpression ::=
    LeftOperand ^ RightOperand

ShiftLeftExpression ::=
    LeftOperand << RightOperand

ShiftRightExpression ::=
    LeftOperand >> RightOperand

Legality Rules

6.5.6:1 A bit expression is an expression that computes a value from two operands using bit arithmetic.

6.5.6:2 A bit and expression is a bit expression that uses bit and arithmetic.

6.5.6:3 A bit or expression is a bit expression that uses bit or arithmetic.

6.5.6:4 A bit xor expression is a bit expression that uses bit exclusive or arithmetic.

6.5.6:5 A shift left expression is a bit expression that uses bit shift left arithmetic.

6.5.6:6 A shift right expression is a bit expression that uses bit shift right arithmetic.

6.5.6:7 The type of the left operand of a bit and expression shall implement the core::ops::BitAnd trait where the type of the right operand is the trait implementation type parameter.

Legality Rules

6.5.10.2:2 The assignee operand of a destructuring assignment corresponds to an assignee pattern according to its kind, as follows:

  • 6.5.10.2:3 A place expression corresponds to an identifier pattern with a unique identifier and without keyword ref, keyword mut, or a bound pattern.

  • 6.5.10.2:4 An underscore expression corresponds to an underscore pattern.

  • 6.5.10.2:5 A tuple expression corresponds to a tuple pattern with all the subexpressions lowered to their corresponding patterns.

  • 6.5.10.2:6 A tuple struct expression corresponds to a tuple struct pattern with all the subexpressions lowered to their corresponding patterns.

  • 6.5.10.2:7 A struct expression corresponds to a struct pattern with all the subexpressions lowered to their corresponding patterns.

  • 6.5.10.2:8 A unit struct expression corresponds to a unit struct pattern.

  • 6.5.10.2:9 A slice expression corresponds to a slice pattern with all the subexpressions lowered to their corresponding patterns.

  • 6.5.10.2:10 A full range expression corresponds to a rest pattern if inside a slice expression, otherwise this is a static error.

6.5.10.2:11 The pattern that corresponds to a destructuring assignment shall be irrefutable.

6.5.10.2:12 A destructuring assignment is equivalent to a block expression of the following form:

  • 6.5.10.2:13 The first statement is a let statement with its pattern equivalent to the lowered assignee pattern and its initialization expression equivalent to the value operand.

  • 6.5.10.2:14 Then each bound identifier of the assignee pattern is an assignment expression used as a statement, as follows:

  • 6.5.10.2:15 The bound identifier becomes the value operand of the new assignment expression, and

  • 6.5.10.2:16 The corresponding expression from the assignee operand of the destructuring assignment becomes the assignee operand of the new assignment expression.

Dynamic Semantics

6.5.10.2:17 The evaluation of a destructuring assignment proceeds as follows:

  1. 6.5.10.2:18 The value operand is evaluated.

  2. 6.5.10.2:19 The assignee operand is evaluated by evaluating its operands in a left-to-right order.

  3. 6.5.10.2:20 Each value denoted by the assignee operand is dropped in left-to-right order, unless the assignee operand denotes an uninitialized binding or an uninitialized field of a binding.

  4. 6.5.10.2:21 The value of the value operand is copied or moved into the place of the assignee operand.

ArrayExpression ::=
    [ ArrayElementExpression? ]
ArrayElementExpression ::=
    ArrayElementConstructor
  | ArrayRepetitionConstructor

ArrayElementConstructor ::=
    ExpressionList

ArrayRepetitionConstructor ::=
    RepeatOperand ; SizeOperand

RepeatOperand ::=
    Operand

SizeOperand ::=
    Operand

Legality Rules

6.8.1:1 An array expression is an expression that constructs an array.

6.8.1:2 An array element constructor is an array expression that lists all elements of the array being constructed.

6.8.1:3 An array repetition constructor is an array expression that specifies how many times an element is repeated in the array being constructed.

6.8.1:4 A repeat operand is an operand that specifies the element being repeated in an array repetition constructor.

6.8.1:5 A size operand is an operand that specifies the size of an array or an array type.

6.8.1:6 An array expression is a value expression.

6.8.1:7 The types of the operands of an array element constructor shall be unifiable.

6.8.1:8 If the size operand is greater than one, then the type of the repeat operand shall implement the core::copy::Copy trait or the repeat operand shall be a path expression resolving to a constant.

6.8.1:9 The type of the size operand shall be type usize.

6.8.1:10 The value of the size operand shall be a constant expression.

6.8.1:11 The type of an array expression is [T; N], where T is the element type and N is the size of the array. The size of an array is determined as follows:

  • 6.8.1:12 If the array expression appears with an array element constructor, then the size is the number of operands in the array element constructor.

  • 6.8.1:13 Otherwise the size is the value of size operand.

6.8.1:14 The value of an array expression is the constructed array.

Dynamic Semantics

6.8.1:15 The evaluation of an array expression with an array element constructor evaluates its operands in left-to-right order.

6.8.1:16 The evaluation of an array expression with an array repetition constructor proceeds as follows:

  1. 6.8.1:17 If the value of the size operand is greater than zero, then:

    1. 6.8.1:18 If the repeat operand denotes a constant, the repeat operand is evaluated once and its value is copied size operand's value times.

    2. 6.8.1:19 Otherwise the repeat operand is evaluated size operand's value times.

  2. 6.8.1:20 Otherwise the repeat operand is evaluated once.

Examples

[1, 2, 3]
["one", "two", "three",]

6.8.1:21 Two dimensional array.

[[0, 0], [0, 1], [1, 0], [1, 1]]

6.8.1:22 An array of nine 42s.

[42; 9]

    Operand

Legality Rules

6.8.2:1 An array index expression is an expression that indexes into an array or a slice.

6.8.2:2 An indexed array operand is an operand which indicates the array or slice being indexed into by an array index expression.

6.8.2:3 An indexing operand is an operand which specifies the index of the array or slice being indexed into by an array index expression.

6.8.2:4 An array index expression is a constant expression if the indexing operand is a constant expression.

6.8.2:5 The type of the indexing operand is the generic parameter of the core::ops::Index implementation of the type of the indexed array operand.

6.8.2:6 If the indexed array operand is evaluated in a value expression context, then

  • 6.8.2:7 The array index expression is a value expression.

  • 6.8.2:9 The type of the array index expression is &T, where T is associated type core::ops::Index::Output.

  • 6.8.2:13 The type of the array index expression is &mut T, where T is the element type of the indexed array operand's type.

6.8.2:14 The value of an array index expression is the indexed memory location.

6.8.2:15 The evaluation of an array index expression proceeds as follows:

  1. 6.8.2:16 The indexed array operand is evaluated.

  2. 6.8.2:17 The indexing operand is evaluated.

  4. 6.8.2:19 Otherwise expression *core::ops::Index::index(&indexed_array_operand, inexing_operand) is evaluated.

6.10:4 A union constructor is a struct expression that constructs a union.

6.10:5 A unit struct constructor is a struct expression that constructs a unit struct.

6.10:7 An indexed initializer is a construct that specifies the index and initial value of a field in a struct expression.

6.10:8 A named initializer is a construct that specifies the name and initial value of a field in a struct expression.

6.10:9 A positional initializer is a construct that specifies the initial value of a field in a struct expression.

6.10:10 A shorthand initializer is a construct that specifies the name of a field in a struct expression.

6.10:11 The construction type indicates the type of the struct being constructed by a struct expression.

6.10:12 A RecordStructConstructor without a RecordStructIndexedFieldInitializerList is a record struct constructor.

6.10:13 A UnitStructConstructor and a RecordStructConstructor without a RecordStructInitializer are unit struct constructors.

6.10:14 A TupleStructConstructor and a RecordStructConstructor without a RecordStructFieldInitializerList are tuple struct constructors.

6.10:15 A RecordStructConstructor with a RecordStructFieldInitializerList is a union constructor.

6.10:16 A struct expression is a value expression.

6.10:17 The type of a struct expression is the construction type.

6.10:18 The value of a struct expression is the struct or union in construction.

6.10:19 The type of a base initializer is the type of its operand. The type of a base initializer shall be the same as the construction type.

Dynamic Semantics

6.10:20 The evaluation of a struct expression evaluates its operands in a left-to-right order.

6.10.1. Record Struct Construction¶

Legality Rules

6.10.1:1 The construction type of a struct constructor shall resolve to a struct type.

6.10.1:2 A named initializer matches a field of the construction type when its identifier and the name of the field are the same.

6.10.1:3 The type of a named initializer and the type of the matched field shall be unifiable.

6.10.1:4 The value of a named initializer is the value of its expression.

6.10.1:5 A named initializer that matches a field is referred to as a matched named initializer.

6.10.1:6 A shorthand initializer matches a field of the construction type when its identifier and the name of the field are the same.

6.10.1:7 The type of a shorthand initializer and the type of the matched field shall be unifiable.

6.10.1:8 The value of a shorthand initializer is the value of its identifier.

6.10.1:9 A shorthand initializer that matches a field is referred to as a matched shorthand initializer.

6.10.1:10 A shorthand initializer is equivalent to a named initializer where both the identifier and the expression of the named initializer denote the identifier of the shorthand initializer.

6.10.1:11 For each field of the construction type, the record struct constructor shall either:

  • 6.10.1:12 Contain a matched named initializer, or

  • 6.10.1:13 Contain a matched shorthand initializer, or

  • 6.10.1:14 Have a RecordStructInitializer with a base initializer or a RecordStructFieldInitializerList with a base initializer.

6.10.1:15 The value of a field of a struct in construction shall be either:

  • 6.10.1:16 The value of a matched named initializer, or

  • 6.10.1:17 The value of a matched shorthand initializer, or

  • 6.10.1:18 The value of the corresponding field of the struct indicated by the base initializer, where the value is either copied or moved.

6.10.2:1 The construction type of a tuple struct constructor shall resolve to a tuple struct type.

6.10.2:3 A positional initializer matches a field of the construction type when the position of the positional initializer and the position of the field in the construction type are the same.

6.11. Call Expressions¶

Syntax

CallExpression ::=
    CallOperand ( ArgumentOperandList? )
CallOperand ::=
    Operand

ArgumentOperandList ::=
    ExpressionList

Legality Rules

6.11:1 A call expression is an expression that invokes a function.

6.11:2 An argument operand is an operand which is used as an argument in a call expression or a method call expression.

6.11:3 A call operand is the function being invoked by a call expression.

6.11:4 A callee type is either a function item type, a function pointer type, or a type that implements any of the core::ops::Fn, core::ops::FnMut, or core::ops::FnOnce traits.

6.11:5 The call operand is subject to auto dereferencing until a callee type is found, otherwise this is a static error.

6.11:6 An adjusted call operand is a call operand with possible auto dereferencing adjustments.

6.11:7 The type of a call expression is the return type of the invoked function or associated type core::ops::FnOnce::Output.

6.11:8 A call expression whose callee type is either an external function item type, an unsafe function item type or an unsafe function pointer type shall require unsafe context.

6.11:9 The value of a call expression is determined as follows:

  • 6.11:10 If the callee type is a function item type or a function pointer type, then the value is the result of invoking the corresponding function with the argument operands.

  • 6.11:11 If the callee type implements the core::ops::Fn trait, then the value is the result of invoking core::ops::Fn::call(adjusted_call_operand, argument_operand_tuple), where adjusted_call_operand is the adjusted call operand, and argument_operand_tuple is a tuple that wraps the argument operands.

  • 6.11:12 If the call operand implements the core::ops::FnMut trait, then the value is the result of invoking core::ops::FnMut::call_mut(adjusted_call_operand, argument_operand_tuple), where adjusted_call_operand is the adjusted call operand, and argument_operand_tuple is a tuple that wraps the argument operands.

  • 6.11:13 If the call operand implements the core::ops::FnOnce trait, then the value is the result of invoking core::ops::FnOnce::call_once(adjusted_call_operand, argument_operand_tuple), where adjusted_call_operand is the adjusted call operand, and argument_operand_tuple is a tuple that wraps the argument operands.

Dynamic Semantics

6.11:14 The evaluation of a call expression proceeds as follows:

  1. 6.11:15 The call operand is evaluated.

  2. 6.11:16 The argument operands are evaluated in left-to-right order.

  3. 6.11:17 If the adjusted call operand is a function item type or function pointer type, then corresponding function is invoked.

  4. 6.11:18 If the type of the call operand implements the core::ops::Fn trait, then core::ops::Fn::call(adjusted_call_operand, argument_operand_tuple) is invoked.

  5. 6.11:19 If the type of the call operand implements the core::ops::FnMut trait, then core::ops::FnMut::call_mut(adjusted_call_operand, argument_operand_tuple) is invoked.

  6. 6.11:20 If the type of the call operand implements the core::ops::FnOnce trait, then core::ops::FnOnce::call_once(adjusted_call_operand, argument_operand_tuple) is invoked.

Undefined Behavior

6.11:21 It is undefined behavior to call a function with an ABI other than the ABI the function was defined with.

Examples

let three: i32 = add(1, 2);

6.12:4 The type of a method call expression is the return type of the invoked method.

6.15. Loop Expressions¶

Syntax

LoopExpression ::=
    Label? LoopContent

LoopContent ::=
    ForLoopExpression
  | InfiniteLoopExpression
  | WhileLetLoopExpression
  | WhileLoopExpression

Legality Rules

6.15:1 A loop expression is an expression that evaluates a block expression continuously as long as some criterion holds true.

6.15:2 An anonymous loop is a loop expression without a label.

6.15:3 A named loop is a loop expression with a label.

6.15:4 The type of a loop expression is determined as follows:

  • 6.15:5 If the loop expression does not contain a break expression, then the type is the never type.

  • 6.15:6 If the loop expression contains at least one break expression, then the type is the unified type of the break types of all break expressions.

6.15:7 The value of a loop expression is determined as follows:

  • 6.15:8 If the loop expression does not contain a break expression, then the value is the unit value.

  • 6.15:9 If the loop expression contains at least one break expression, then the value is the break value of the break expression that broke out of the loop expression.

Dynamic Semantics

6.15:10 A loop expression is terminated when its block expression is no longer evaluated.

6.15.1. For Loops¶

Syntax

ForLoopExpression ::=
    for Pattern in SubjectExpression BlockExpression

Legality Rules

6.15.1:1 A for loop expression is a loop expression that continues to evaluate its block expression as long as its subject expression yields a value.

6.15.1:2 The type of a subject expression shall implement the core::iter::IntoIterator trait.

Dynamic Semantics

6.15.1:3 The evaluation of a for loop expression of the form

'label: for pattern in subject_expression {
    /* loop body */
}

6.15.1:4 is equivalent to the evaluation of the following block expression:

{
    let result =
        match core::iter::IntoIterator::into_iter
            (subject_expression)
    {
        mut iter => 'label: loop {
            let mut next_value;
            match core::iter::Iterator::next(&mut iter) {
                Option::Some(value) => next_value = value,
                Option::None => break
            };
            let pattern = next_value;
            let () = { /* loop body */ };
        }
    };
    result
}

Examples

let favorite_fruits = &["apples", "pears", "strawberries"];

for fruit in favorite_fruits {
    println!("I like eating {}.", fruit);
}

Syntax

6.15.2:1 An infinite loop expression is a loop expression that continues to evaluate its block expression indefinitely unless terminated with a break expression or a return expression.

Dynamic Semantics

6.15.2:2 The evaluation of an infinite loop expression proceeds as follows:

  1. 6.15.2:3 The block expression is evaluated.

  2. 6.15.2:4 Control restarts the evaluation of the infinite loop expression.

Examples

Syntax

WhileLoopExpression ::=

IterationExpression ::=

Legality Rules

6.15.3:1 A while loop expression is a loop expression that continues to evaluate its block expression as long as its iteration expression holds true.

6.15.3:2 An iteration expression is an expression that provides the criterion of a while loop expression.

6.15.3:3 The type of an iteration expression shall be type bool.

Dynamic Semantics

6.15.3:4 The evaluation of a while loop expression proceeds as follows:

  1. 6.15.3:5 The iteration expression is evaluated.

  2. 6.15.3:6 If the iteration expression evaluated to true, then:

    1. 6.15.3:7 The block expression is evaluated.

    2. 6.15.3:8 Control restarts the evaluation of the while loop expression.

6.15.6:2 A break expression shall appear within a loop expression.

6.15.6:3 The label indication of a break expression shall resolve to the label of an enclosing named loop.

6.15.6:4 A break expression without a label indication is associated with the innermost enclosing loop expression.

6.15.6:5 A break expression with a label indication is associated with a named loop whose label is indicated by the label indication.

6.15.6:6 A break expression shall have an operand only when it is associated with an infinite loop.

6.15.6:7 The type of a break expression is the never type.

6.15.6:8 Break type is the type of the operand of a break expression.

6.15.6:9 The break type is determined as follows:

  • 6.15.6:10 If the break expression lacks an operand, then the break type is the unit type.

  • 6.15.6:11 If the break expression has an operand, then the break type is the type of its operand.

6.15.6:12 Break value is the value of the operand of a break expression.

6.15.6:13 The break value is determined as follows:

  • 6.15.6:14 If the break expression lacks an operand, then the break value is the unit value.

  • 6.15.6:15 If the break expression has an operand, then the break value is the value of its operand.

Dynamic Semantics

6.15.6:16 The evaluation of a break expression proceeds as follows:

  1. 6.15.6:17 The operand is evaluated.

  2. 6.15.6:18 All enclosing loop expressions upto and including the associated loop expression are terminated.

Examples

6.15.6:19 The following break expression terminates both the inner and the outer loop.

'outer: loop {
    'inner: loop {
        break 'outer;
    }
}

6.15.7. Continue Expressions¶

Syntax

ContinueExpression ::=
    continue LabelIndication?

Legality Rules

6.15.7:1 A continue expression shall appear within a loop expression.

6.15.7:2 A continue expression without a label indication is associated with the innermost enclosing loop expression.

6.15.7:3 A continue expression with a label indication is associated with a named loop whose label is indicated by the label indication.

6.15.7:4 The type of a continue expression is the never type.

6.15.7:5 The value of a continue expression is the unit value.

Dynamic Semantics

6.15.7:6 The evaluation of a continue expression proceeds as follows:

  1. 6.15.7:7 If the continue expression appears with a label indication, then all enclosing loop expressions upto and including the associated loop expression are terminated.

  2. 6.15.7:8 The evaluation of the associated loop expression is restarted.

Examples

6.15.7:9 The following continue expression terminates and restarts game_loop.

'game_loop: loop {
    if is_paused() {
        continue;
    }
    . . .
}

6.16. Range Expressions¶

Syntax

RangeExpression ::=
    RangeFromExpression
  | RangeFromToExpression
  | RangeFullExpression
  | RangeInclusiveExpression
  | RangeToExpression
  | RangeToInclusiveExpression

RangeFromExpression ::=
    RangeExpressionLowBound ..

RangeFromToExpression ::=
    RangeExpressionLowBound .. RangeExpressionHighBound

RangeFullExpression ::=
    ..

RangeInclusiveExpression ::=
    RangeExpressionLowBound ..= RangeExpressionHighBound

RangeToExpression ::=
    .. RangeExpressionHighBound

RangeToInclusiveExpression ::=
    ..= RangeExpressionHighBound

RangeExpressionLowBound ::=
    Operand

RangeExpressionHighBound ::=
    Operand

Legality Rules

6.16:1 A range expression is an expression that constructs a range.

6.16:2 A range expression low bound is an operand that specifies the start of a range.

6.16:3 A range expression high bound is an operand that specifies the end of a range.

6.16:4 If a range expression has two operands, then the types of the operands shall be unifiable.

6.16:5 A range-from expression is a range expression that specifies an included range expression low bound.

6.16:6 The type of a range-from expression is core::ops::RangeFrom.

6.16:7 The value of a range-from expression is core::ops::RangeFrom { start: range_expression_low_bound }.

6.16:8 A range-from-to expression is a range expression that specifies an included range expression low bound and an excluded range expression high bound.

6.16:9 The type of a range-from-to expression is core::ops::Range.

6.16:10 The value of a range-from-to expression is core::ops::Range { start: range_expression_low_bound, end: range_expression_high_bound }.

6.16:11 A range-full expression is a range expression that covers the whole range of a type.

6.16:12 The type of a range-full expression is core::ops::RangeFull.

6.16:13 The value of a range-full expression is core::ops::RangeFull {}.

6.16:14 A range-inclusive expression is a range expression that specifies an included range expression low bound and an included range expression high bound.

6.16:15 The type of a range-inclusive expression is core::ops::RangeInclusive.

6.16:16 The value of a range-inclusive expression is core::ops::RangeInclusive::new(range_expression_low_bound, range_expression_high_bound).

6.16:17 A range-to expression is a range expression that specifies an excluded range expression high bound.

6.16:18 The type of a range-to expression is core::ops::RangeTo.

6.16:19 The value of a range-to expression is core::ops::RangeTo { end: range_expression_high_bound }.

6.16:20 A range-to-inclusive expression is a range expression that specifies an included range expression high bound.

6.16:21 The type of a range-to-inclusive expression is core::ops::RangeToInclusive.

6.16:22 The value of a range-to-inclusive expression is core::ops::RangeToInclusive { end: range_expression_high_bound }.

Dynamic Semantics

6.16:23 The evaluation of a range expression evaluates its operands in left-to-right order.

Examples

1 ..
42 .. 86
..
dawn ..= dusk
..= 5

6.17. If and If Let Expressions¶

6.17.1. If Expressions¶

Syntax

IfExpression ::=
    if SubjectExpression BlockExpression ElseExpression?

ElseExpression ::=
    else (BlockExpression | IfExpression | IfLetExpression)

Legality Rules

6.17.1:1 An if expression is an expression that evaluates either a block expression or an else expression depending on the value of its subject expression.

6.17.1:2 An else expression is an expression that represents either a block expression, an if expression, or an if let expression.

6.17.1:3 The type of the subject expression of an if expression shall be type bool.

6.17.1:4 The type of an if expression is the type of its block expression.

6.17.1:5 The value of an if expression is the value of its block expression.

6.17.1:6 The type of an else expression is the type of its block expression, if expression, or if let expression.

6.17.1:7 The value of an else expression is the value of its block expression, if expression, or if let expression.

6.17.1:8 The type of an if expression and the type of an else expression shall be unifiable.

Dynamic Semantics

6.17.1:9 The evaluation of an if expression proceeds as follows:

  1. 6.17.1:10 The subject expression is evaluated.

  2. 6.17.1:11 If the subject expression evaluated to true, then the block expression is evaluated.

  3. 6.17.1:12 If the subject expression evaluated to false and the if expression has an else expression, then the else expression is evaluated.

6.17.1:13 The evaluation of an else expression evaluates its block expression, if expression, or if let expression.

Examples

if age <= 14 {
    println!("child");
} else if age <= 24 {
    println!("youth");
} else if age <=64 {
    println!("adult");
} else {
    println!("senior");
}

6.17.2. If Let Expressions¶

Syntax

IfLetExpression ::=
    if let Pattern = SubjectLetExpression BlockExpression ElseExpression?

Legality Rules

6.17.2:1 An if let expression is an expression that evaluates either a block expression or an else expression depending on whether its pattern can be matched against its subject let expression.

6.17.2:2 The type of an if let expression is the type of its block expression.

6.17.2:3 The value of an if let expression is the value of its block expression.

Dynamic Semantics

6.17.2:4 The evaluation of an if let expression of the form

if let pattern = subject_let_expression {
    /* body */
}

6.17.2:5 is equivalent to the evaluation of the following match expression:

match subject_let_expression {
    pattern => { /* body */ },
    _ => ()
}

6.17.2:6 The evaluation of an if let expression of the form

if let pattern = subject_let_expression {
    /* body */
} else {
    /* else */
}

6.17.2:7 is equivalent to the evaluation of the following match expression:

match subject_let_expression {
    pattern => { /* body */ },
    _ => { /* else */ }
}

Examples

let dish = ("Ham", "Eggs");

if let ("Ham", side) = dish {
    println!("Ham is served with {}", side);
}

6.18. Match Expressions¶

Syntax

MatchExpression ::=
    match SubjectExpression {
      InnerAttributeOrDoc*
      MatchArmList?
    }
MatchArmList ::=
    IntermediateMatchArm* FinalMatchArm

IntermediateMatchArm ::=
    MatchArmMatcher => ( ExpressionWithBlock ,? | ExpressionWithoutBlock , )

FinalMatchArm ::=
    MatchArmMatcher => Operand ,?

MatchArmMatcher ::=
    OuterAttributeOrDoc* Pattern MatchArmGuard?

MatchArmGuard ::=
    if Operand

Legality Rules

6.18:1 A match expression is an expression that tries to match one of its multiple patterns against its subject expression and if it succeeds, evaluates an operand.

6.18:2 A match arm is a construct that consists of a match arm matcher and a match arm body.

6.18:3 An intermediate match arm is any non-final match arm of a match expression.

6.18:4 A final match arm is the last match arm of a match expression.

6.18:5 A match arm matcher is a construct that consists of a pattern and a match arm guard.

6.18:6 A match arm body is the operand of a match arm.

6.18:7 A match arm guard is a construct that provides additional filtering to a match arm matcher.

6.18:8 A match expression is a place expression when its subject expression is a place expression. When a match expression is a place expression, the value produced by evaluating its subject expression is copied or moved.

6.18:9 A match expression is a value expression when its subject expression is a value expression. When the match expression is a value expression, the value produced by evaluating its subject expression is captured in a temporary.

6.18:10 The type of the subject expression and the types of all patterns of all match arm matchers shall be unifiable.

6.18:11 The type of the operand of a match arm guard shall be type bool.

6.18:12 The types of all match arm bodies shall be unifiable.

6.18:13 The type of a match expression is the unified type of the types of the operands of all match arms.

6.18:14 A match arm is selected when its pattern matches the subject expression and its match arm guard (if any) evaluates to true.

6.18:15 Match arm selection happens in declarative order.

6.18:16 The patterns of all match arms taken together shall exhaustively match the subject expression's type.

6.18:17 The value of a match expression is the value of the operand of the selected match arm.

Dynamic Semantics

6.18:18 The evaluation of a match expression proceeds as follows:

  1. 6.18:19 The subject expression is evaluated.

  2. 6.18:20 Each match arm is evaluated in declarative order as follows:

    1. 6.18:21 The match arm marcher of the match arm is evaluated.

    2. 6.18:22 If the match arm marcher succeeds, then

      1. 6.18:23 The operand of the match arm is evaluated.

      2. 6.18:24 Control stops the evaluation of the match expression.

    3. 6.18:25 Otherwise control proceeds with the evaluation of the next match arm.

6.18:26 The evaluation of a match arm marcher proceeds as follows:

  1. 6.18:27 The pattern of the match arm marcher is evaluated.

  2. 6.18:28 If the pattern succeeds, then

    1. 6.18:29 If the match arm marcher has a match arm guard, then

      1. 6.18:30 The match arm guard is evaluated.

      2. 6.18:31 If the match arm guard evaluates to true, then the match arm marcher succeeds.

    2. 6.18:32 Otherwise the match arm marcher fails.

  3. 6.18:33 Otherwise the match arm marcher fails.

6.18:34 The evaluation of a match arm guard evaluates its operand. A match arm guard evaluates to true when its operand evaluates to true, otherwise it evaluates to false.

Examples

fn quantify(number_of_things: i32) {
    match number_of_things {
        0 | 1 => println!("not many"),
        2 ..= 9 => println!("a few"),
        _ if number_of_things < 0 => println!("you owe me"),
        _ => println!("lots")
    }
}

6.19. Return Expressions¶

Syntax

ReturnExpression ::=
    return Expression?

Legality Rules

6.19:1 A return expression is an expression that optionally yields a value and causes control flow to return to the caller.

6.19:2 A return expression shall appear within a control flow boundary.

6.19:3 The type of a return expression is determined as follows:

  • 6.19:4 If the return expression has an operand, then the type is the type of the operand.

  • 6.19:5 If the return expression does not have an operand, then the type is the never type.

6.19:6 The value of a return expression is determined as follows:

  • 6.19:7 If the return expression has an operand, then the value is the value of the operand.

  • 6.19:8 If the return expression does not have an operand, then the value is the unit value.

Dynamic Semantics

6.19:9 The evaluation of a return expression proceeds as follows:

  1. 6.19:10 If the return expression has an operand, then

    1. 6.19:11 The operand is evaluated.

    2. 6.19:12 The value of the operand is moved into the designated output location of the enclosing control flow boundary.

  2. 6.19:13 Control destroys the current activation frame.

  3. 6.19:14 Control is transferred to the caller frame.

Examples

fn max(left: i32, right: i32) -> i32 {
    if left > right {
        return left;
    }
    return right;
}

6.20. Await Expressions¶

Syntax

AwaitExpression ::=
    FutureOperand . await

FutureOperand ::=
    Operand

Legality Rules

6.20:1 An await expression is an expression that polls a future, suspending the execution of the future until the future is ready.

6.20:2 A future operand is an operand whose future is being awaited by an await expression.

6.20:3 An await expression shall appear within

  • 6.20:4 An async block expression.

  • 6.20:5 An async function.

6.20:6 The type of a future operand shall implement the core::future::Future trait.

6.20:7 The type of an await expression is <_ as core::future::Future>::Output.

6.20:8 The value of an await expression is the value held by core::task::Poll::Ready.

Dynamic Semantics

6.20:9 The evaluation of an await expression proceeds as follows:

  1. 6.20:10 The future operand is evaluated to a temporary.

  2. 6.20:11 The temporary is pinned using core::pin::Pin::new_unchecked.

  3. 6.20:12 The pinned temporary is polled using core::future::Future::poll, passing in the core::task::Context of the current task.

  4. 6.20:13 If core::future::Future::poll returns core::task::Poll::Pending, then the current future yields.

  5. 6.20:14 If core::future::Future::poll returns core::task::Poll::Ready, then

    1. 6.20:15 The value held within is unwrapped.

    2. 6.20:16 Control stops the evaluation of the await expression.

Examples

let future = async { expensive_function() };
future.await;

6.22. Capturing¶

Legality Rules

6.22:1 A capturing expression is either an async block expression or a closure expression.

6.22:2 A capture target is either a binding or a field of a binding.

6.22:3 The capturing environment of a capturing expression consists of all capture targets that are defined outside the capture expression.

6.22:4 Capturing is the process of saving the capture targets of a capturing expression's capturing environment.

6.22:5 A capturing target requires capturing when it is part of the capturing expression's capture environment and it is used by the capturing expression. Such a capturing target is said to be captured.

6.22:6 Capture mode is the mechanism by which a capture target is captured.

6.22:7 The capture mode is determined based on the use of the capture target within the capturing expression, as follows:

  1. 6.22:8 If the capturing expression is subject to keyword move, then

    1. 6.22:9 If the type of the capture target is a by copy type, then the capture mode is by copy.

    2. 6.22:10 Otherwise the capture mode is by move.

  2. 6.22:11 Otherwise the capture mode is determined based on the following precedence:

    1. 6.22:12 By immutable borrow mode.

    2. 6.22:13 By unique immutable borrow mode, if the capture target is a mutable reference that is being modified.

    3. 6.22:14 By mutable borrow mode.

    4. 6.22:15 If the type of the capture target is a by copy type, then the capture mode is by copy, otherwise it is by move.

6.22:16 A tool selects the first capture mode that is compatible with the use of the capture target.
